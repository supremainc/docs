## Windows Server LTS 릴리스 사용

지원되지 않거나 오래된 Windows 버전에서 BioStar X를 실행하면 패치가 적용되지 않은 취약점에 시스템이 노출됩니다. Microsoft는 보안 업데이트, 안정성 수정 및 10년 이상(5년 기본 지원 + 5년 연장 지원) 지원을 보장하는 Windows Server 장기 지원(LTS) 버전을 제공합니다. LTS 릴리스를 선택하면 BioStar X 서버가 가능한 가장 긴 보안 지원 기간을 확보하고, 기능 업데이트로 인한 시스템 중단을 최소화할 수 있습니다.다.

:::note

**관리자를 위한 요약**

- 반드시 **Windows Server LTS 에디션**(2019 또는 2022)에 BioStar X를 배포하세요.

- 설치 후 운영체제 버전을 **확인**하세요.

- 지원 종료일 이전에 **업그레이드를 계획**하세요.

- **자동 업데이트를 활성화**하거나 WSUS/SCCM에 연결하세요.

- 지원되지 않는 소프트웨어를 실행하지 않도록 **제품 수명 주기를 문서화**하세요.


:::


### 지원 버전 확인

Microsoft에서 활성 지원하는 LTS Windows Server 에디션(2026년 5월 기준):

- Windows Server 2019 (LTS) — 기본 지원: 2024년 1월까지 / 연장 보안 지원: 2029년 1월까지

- Windows Server 2022 (LTS) — 기본 지원: 2026년 10월까지 / 연장 보안 지원: 2031년 10월까지

- Windows Server 2025 (LTS) — 기본 지원: 2029년 11월까지 / 연장 보안 지원: 2034년 11월까지

:::caution

Semi-Annual Channel(SAC) 릴리스는 수명이 짧고 BioStar X와 같은 프로덕션 워크로드에는 적합하지 않으므로 피하십시오.

:::


### 현재 OS 버전 확인

- <Kbd>Win + R</Kbd> 키를 누른 후 `winver`를 입력하세요. <Kbd>Enter</Kbd> 키를 누르면 에디션 및 버전이 표시됩니다.

- PowerShell에서 다음 명령어를 실행하여 OS 이름과 버전을 확인할 수도 있습니다:

   ```powershell
   systeminfo | findstr /B /C:"OS Name" /C:"OS Version"
   ```


### 업그레이드 계획

- Windows Server 2016 버전 이하를 실행 중이라면 마이그레이션을 계획해야 합니다.

- 2019 또는 2022 LTS 버전으로 업그레이드하면 보안 업데이트 및 최신 TLS 프로토콜(TLS 1.2/1.3)과의 호환성이 보장됩니다.

:::note

**모범 사례**

지원되는 운영체제로 새 가상머신(VM)이나 서버를 구축하세요. BioStar X를 새로 설치한 후 구성 및 데이터베이스를 마이그레이션하세요. 이를 통해 다운타임과 위험을 최소화할 수 있습니다.

:::


### Windows Update 구성 확인

보안 패치를 위한 **자동 업데이트**를 활성화하세요. 또는 Windows Server Update Services(WSUS) 또는 System Center Configuration Manager(SCCM)를 사용하는 경우 호스트가 올바르게 등록되어 있는지 확인하세요.

## 최신 Windows 업데이트 및 보안 패치 적용

프로덕션 배포 전에 최신 Windows 업데이트와 보안 패치를 적용하여 알려진 취약점으로부터 시스템을 보호하세요.

패치되지 않은 Windows 서버는 랜섬웨어, 원격 코드 실행, 권한 상승 익스플로잇(Exploit)의 **주요 공격 경로**입니다. 최신 보안 업데이트를 적용하면 BioStar X 호스트가 모든 **알려진 취약점**으로부터 보호된 상태로 라이프사이클을 시작할 수 있습니다.


### 현재 패치 수준 확인

- 최신 패치 상태를 확인하려면 **설정** → **Windows 업데이트**를 열고 **업데이트 확인**을 클릭하세요.

- PowerShell에서 최근 설치된 패치 10개를 확인하려면 다음 명령어를 실행하세요.

   ```powershell
   Get-HotFix | Sort-Object -Descending InstalledOn | Select-Object -First 10
   ```

:::caution

최신 패치가 30일 이상 지났다면 즉시 업데이트를 진행하세요.

:::

**업데이트 프로세스**

1. Windows Update 수동 실행
2. 모든 보안 및 중요 업데이트 설치 (서버가 아직 프로덕션 상태가 아니더라도 미루지 말 것)
3. 필요시 재부팅
4. "최신 상태입니다"가 표시될 때까지 업데이트 확인 반복

**지속적인 패치 전략**

- **월별 "패치 화요일"**: Microsoft는 매월 두 번째 화요일에 새 패치를 배포합니다.
- 스테이징 환경이 있다면 테스트 후 배포에서 1–2주 이내에 패치를 적용합니다.
- 즉시 패치할 수 없는 경우 **보완적 제어**를 활성화합니다: 방화벽 제한, 인터넷 접근 차단, 패치 전까지 강화 모니터링.

#### 3.1.3. 불필요한 Windows 기능 제거 또는 비활성화

**중요한 이유:**  
활성화된 모든 Windows 기능은 서버의 **공격 표면을 증가**시킵니다. 전용 BioStar X 호스트에서는 Print Spooler, Fax, IIS 등의 서비스가 거의 필요 없으며, 이는 심각한 취약점(예: PrintNightmare)의 이력을 가지고 있습니다.

**단계별 조치:**

1. **서버 관리자** → 관리 → 역할 및 기능 제거
2. 설치된 역할(예: 파일 서비스, IIS, 인쇄 서비스)을 검토하고 BioStar X에 필요하지 않은 역할을 제거합니다.
3. **.NET Framework 기능** 및 관리 워크플로에 필요한 **데스크탑 환경**만 유지합니다.
4. `services.msc`를 열고 불필요한 서비스(Fax, Remote Registry, Bluetooth, SMBv1)를 **사용 안 함**으로 설정합니다.

👍 **모범 사례**: 호스트를 BioStar X **전용**으로 유지합니다. 추가 워크로드 없음.

---

#### 3.1.4. Windows Defender 바이러스 백신 활성화 및 최신 상태 유지

**중요한 이유:**  
Defender(기본 제공 AV)는 악성 코드 및 랜섬웨어로부터 보호합니다. 비활성화하면 호스트가 노출됩니다.

**단계별 조치:**

1. **Windows 보안** → 바이러스 및 위협 방지 열기
2. **실시간 보호**가 활성화되어 있는지 확인
3. **클라우드 기반 보호** 및 **자동 샘플 제출**이 활성화되어 있는지 확인
4. 바이러스 정의를 최신 상태로 유지 (일반적으로 Windows Update를 통해 자동으로 수행)

> ⚠️ 타사 AV를 사용하는 경우 충돌을 방지하기 위해 **하나의 AV 솔루션**만 실행되고 있는지 확인합니다.

#### 3.1.5. 물리적 도난 방지를 위한 BitLocker 활성화

**중요한 이유:**  
공격자가 서버에 물리적으로 접근하면(예: 디스크 도난, VM 스냅샷, 백업 미디어) 암호화되지 않은 드라이브에서 자격 증명, 로그, 생체 정보 등 민감한 데이터에 직접 접근할 수 있습니다. BitLocker는 미사용 데이터 보호를 위한 전체 디스크 암호화를 제공합니다.

**권고 사항:**

- 다음에 BitLocker를 활성화합니다:
  - OS 드라이브(C:)
  - BioStar X 데이터베이스, 로그, 백업을 저장하는 모든 데이터 드라이브
- 더 강력한 보호를 위해 가능한 경우 TPM + PIN 사용

**단계별 조치 (GUI):**

1. `제어판 → BitLocker 드라이브 암호화` 열기
2. 각 드라이브에 대해 "BitLocker 켜기" 클릭
3. 인증 방법 선택: TPM(기본값) 또는 TPM + PIN(더 높은 보안에 권장)
4. 복구 키 저장: Active Directory/Azure AD(선호) 또는 보안 볼트(동일 서버에 저장 금지)
5. 암호화 모드: XTS-AES 256비트(권장)
6. 암호화 시작

**모니터링:**

```powershell
Get-BitLockerVolume
```

> ⚠️ BitLocker는 **미사용 데이터만** 보호합니다. 데이터베이스 암호화 또는 TLS 암호화를 대체하지 않습니다.

#### 3.1.6. 안전한 인증 정책 시행

**중요한 이유:**  
약한 로컬 계정이나 캐시된 자격 증명은 서버를 무차별 대입 공격이나 권한 상승에 취약하게 만듭니다.

**단계별 조치:**

1. **로컬 보안 정책**(`secpol.msc`) 열기
2. 암호 정책 구성:
   - 최소 길이: 12자 이상
   - 복잡성: 활성화
   - 최대 사용 기간: 90일
   - 잠금 임계값: 5회 시도
3. 기본 "Administrator" 계정 비활성화 또는 이름 변경
4. RDP 또는 원격 관리 접근에 대해 MFA 요구

---

#### 3.1.7. 보안 기준선을 위한 그룹 정책 설정 시행

**중요한 이유:**  
Microsoft의 **Security Compliance Toolkit(SCT)**은 보안 기준선에 맞추어진 다운로드 가능한 **그룹 정책 개체(GPO)**를 제공합니다. 이 기준선은 구성 오류 위험을 줄이고 BioStar X가 강화된 Windows OS에서 실행되도록 합니다.

**단계별 조치:**

1. [Microsoft Security Compliance Toolkit](https://learn.microsoft.com/en-us/windows/security/operating-system-security/device-management/windows-security-configuration-framework/security-compliance-toolkit-10) 다운로드
2. OS 버전에 맞는 기준선 선택 (Windows Server 2019 또는 2022 Security Baseline)
3. **그룹 정책 관리 콘솔(GPMC)**에 기준선 GPO 가져오기
4. BioStar X 서버 OU에 기준선 GPO 연결
5. 중요 설정 검토: 계정 잠금, 감사 로깅, 네트워크 보안(SMB 서명, LDAP 서명), LSA 보호
6. **스테이징에서 먼저 테스트** 후 BioStar X 호환성 확인 후 프로덕션에 배포

#### 3.1.8. 필요하지 않은 경우 원격 데스크톱 비활성화(필요한 경우 강화)

**중요한 이유:**  
RDP는 무차별 대입 공격, 자격 증명 도용의 일반적인 공격 벡터입니다.

**필요하지 않은 경우:**
- `시스템 속성 → 원격 → "원격 연결 허용 안 함"`으로 비활성화

**필요한 경우:**
- VPN 또는 LAN으로만 제한
- NLA(네트워크 수준 인증) 요구
- 기본 RDP 포트 변경 (3389 → 사용자 지정)
- 이벤트 로그 및 알림으로 모니터링
