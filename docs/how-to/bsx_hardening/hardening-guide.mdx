## 1. 소개

본 Hardening Guide의 목적은 관리자, 보안 엔지니어, 배포 팀이 BioStar X Access Control System의 보안 수준을 향상시키기 위한 실질적인 권고 사항을 제공하는 것입니다.

보안 강화(Hardening)는 불필요한 기능을 비활성화하고, 안전한 기본값을 적용하며, 공인된 사이버 보안 모범 사례에 맞춰 시스템 구성을 정렬함으로써 공격 표면을 최소화하는 과정입니다. 본 가이드는 서버, 데이터베이스, API, 클라이언트 애플리케이션을 포함한 시스템 구성 요소 전반에 걸쳐 시스템을 보안하는 데 필요한 단계를 설명합니다.

본 가이드는 시스템의 설치, 구성, 유지 관리를 담당하는 IT 및 보안 전문가를 대상으로 하며, 기본 보안 조치와 고급 보안 조치를 모두 다루어 조직이 위협 모델 및 컴플라이언스 요건에 따라 강화 접근 방식을 조정할 수 있도록 합니다.

이 권고 사항들을 따르면 무단 접근, 악성 코드, 데이터 침해에 대한 시스템 탄력성이 크게 향상됩니다. 그러나 어떠한 강화 절차도 완전한 보호를 보장할 수 없습니다. 따라서 이러한 방법론은 지속적인 모니터링, 패치 관리, 수립된 인시던트 대응 프로세스로 보완되어야 합니다.

본 가이드는 OWASP Top 10, CIS Controls, NIST Cybersecurity Framework 등 업계 표준에 맞추어 보안 관행이 세계적으로 인정된 벤치마크와 일치하도록 합니다.

본 가이드는 세 가지 주요 부분으로 구성됩니다:

- **보안 접근 제어의 핵심 원칙** – 강화 프로세스를 안내하는 기본 보안 개념 개요.
- **서버 강화** – BioStar X를 호스팅하는 운영 체제, 네트워크, 물리적 환경 보안 권고 사항.
- **애플리케이션 강화** – BioStar X의 내장 보안 기능을 최대한 활용하는 방법. 이 부분은 두 가지 하위 섹션으로 나뉩니다:
  - **시스템 보안** – 전반적인 보안 향상을 위한 BioStar X 기능 구성 방법.
  - **개인 데이터 보안** – 개인 및 민감 데이터 보호를 위한 BioStar X 기능 활용 방법.

## 2. 보안 접근 제어의 핵심 원칙

접근 제어 시스템의 보안 강화는 단순히 기술적인 설정에 관한 것이 아닙니다. 이는 사람, 프로세스, 기술 전반에 걸쳐 검증된 원칙을 일관되게 적용하는 것입니다. BioStar X는 설계상 이러한 원칙을 지원하지만, 관리자가 올바르게 구성하고 시행해야 합니다.

### 2.1. 최소 권한 원칙 (PoLP)

사용자와 서비스에는 작업 수행에 **필요한 권한만** 부여해야 합니다. 그 이상도 이하도 안 됩니다.

- 관리자 계정은 일상적인 운영이 아닌 구성 및 시스템 변경에만 사용해야 합니다.
- 운영자는 도어, 사용자, 이벤트를 관리할 수 있지만 보안 정책은 변경할 수 없어야 합니다.
- 감사자는 데이터를 변경할 수 없는 완전한 읽기 전용 가시성을 가져야 합니다.

**중요한 이유:**  
권한을 제한하면 계정이 침해되더라도 피해를 최소화할 수 있습니다. 공격자가 운영자 계정을 탈취하더라도 암호화 키를 변경하거나 로그를 비활성화할 수 없습니다.

### 2.2. 직무 분리

중요한 책임은 **여러 역할에 분산**되어야 하므로 어느 한 사람도 시스템에 대한 무제한적인 통제권을 가질 수 없습니다.

- 예: 한 관리자는 시스템 구성을 관리하고, 다른 운영자는 일상적인 접근 요청을 처리합니다.
- 감사자는 독립적으로 변경 사항을 확인하고 로그를 모니터링합니다.

**중요한 이유:**  
내부자 남용을 방지하고 **견제와 균형**을 보장합니다. 한 계정이 침해되더라도 공격자는 전체 시스템을 통제할 수 없습니다.

### 2.3. 심층 방어 (Defense in Depth)

어떠한 단일 보안 조치도 완벽하지 않습니다. 대신, 하나가 실패하더라도 다른 보안 계층이 시스템을 보호할 수 있도록 여러 보안 제어 계층을 적용해야 합니다.

- **네트워크 계층**: 방화벽, VLAN 세분화.
- **시스템 계층**: 안전한 OS 기준선, 패치, 안티바이러스.
- **애플리케이션 계층**: 강력한 비밀번호, MFA, 암호화.
- **모니터링 계층**: 로그, SIEM, 알림.

**중요한 이유:**  
공격자는 종종 취약한 연결 고리를 찾습니다. 심층 방어는 단일 실패 지점이 없도록 보장합니다.

### 2.4. 제로 트러스트 아키텍처 개념

어떠한 장치, 사용자, 네트워크도 기본적으로 신뢰해서는 안 됩니다 — **"절대 신뢰하지 말고, 항상 검증하라."**

- 모든 로그인은 인증을 요구해야 합니다(가능한 경우 MFA).
- 장치-서버 간 통신은 항상 암호화되어야 합니다.
- 내부 트래픽도 외부 트래픽처럼 세분화하고 모니터링해야 합니다.

**중요한 이유:**  
현대의 위협은 종종 네트워크 내부에서 발생합니다(손상된 노트북, 피싱, 악의적인 내부자). 제로 트러스트는 "신뢰할 수 있는 경계"에 대한 의존을 줄이고 지속적인 검증을 시행합니다.

이 네 가지 원칙은 **본 Hardening Guide의 기반**을 형성합니다. 이후 장에서의 모든 구성 권고 사항(포트 설정부터 디렉터리 통합까지)은 **PoLP, 직무 분리, 심층 방어, 제로 트러스트**의 실질적인 적용입니다.

## 3. BioStar X 서버 강화 가이드

본 섹션에서는 BioStar X를 호스팅하는 Windows 환경을 보안하기 위한 권고 사항을 제공합니다. 취약점을 최소화하고, 공격 표면을 줄이며, 조직 보안 정책 준수를 보장하기 위한 시스템 수준, 네트워크 수준, 애플리케이션 수준 제어를 다룹니다.

**범위:**
- BioStar X를 실행하는 모든 Windows Server(베어 메탈 또는 VM)에 적용됩니다.
- 시스템 구성, 네트워크 보호, 접근 제어, 모니터링, 패치를 다룹니다.
- 시스템 관리자, 보안 엔지니어, 배포 팀을 대상으로 합니다.

### 3.1. 운영 체제 구성

#### 3.1.1. Microsoft가 지원하는 Windows Server LTS 릴리스 사용

**중요한 이유:**  
지원되지 않거나 오래된 Windows 버전에서 BioStar X를 실행하면 패치되지 않은 취약점에 노출됩니다. Microsoft는 10년 이상(주류 지원 5년 + 연장 지원 5년)의 보안 업데이트, 안정성 수정, 지원을 보장하는 Windows Server의 장기 서비스(LTS) 릴리스를 제공합니다. LTS 릴리스를 선택하면 BioStar X 서버가 파괴적인 기능 업데이트 없이 가능한 가장 긴 보안 지원 기간을 갖출 수 있습니다.

**지원되는 버전 식별**

작성 시점 기준, Microsoft에서 활성 지원하는 LTS Windows Server 에디션:

- Windows Server 2019 (LTS) — 주류 지원: 2024년 1월까지 / 연장 보안 지원: 2029년 1월까지
- Windows Server 2022 (LTS) — 주류 지원: 2026년 10월까지 / 연장 보안 지원: 2031년 10월까지
- Windows Server 2025 (LTS) — 주류 지원: 2029년 11월까지 / 연장 보안 지원: 2034년 11월까지

> ⚠️ Semi-Annual Channel(SAC) 릴리스는 수명이 짧고 BioStar X와 같은 프로덕션 워크로드에는 적합하지 않으므로 피하십시오.

**현재 OS 버전 확인**

- `Win + R`을 누르고 `winver`를 입력한 후 Enter를 누르면 에디션 및 버전이 표시됩니다.
- 또는 PowerShell에서 다음을 실행합니다:

```powershell
systeminfo | findstr /B /C:"OS Name" /C:"OS Version"
```

**업그레이드 계획**

- Windows Server 2016 이하를 실행 중인 경우 마이그레이션을 계획해야 합니다.
- 2019 또는 2022 LTS로 업그레이드하면 보안 업데이트 및 최신 TLS 프로토콜(TLS 1.2/1.3)과의 호환성이 보장됩니다.

👍 **모범 사례**: 지원되는 OS로 새 VM이나 서버를 구축하고 BioStar X를 새로 설치한 후 구성과 데이터베이스를 마이그레이션합니다.

**Windows Update 구성 확인**

- 보안 패치를 위한 자동 업데이트를 활성화하거나
- WSUS(Windows Server Update Services) 또는 SCCM을 사용하는 경우 호스트가 올바르게 등록되어 있는지 확인합니다.

**요약**

- 항상 **Windows Server LTS 에디션**(2019 또는 2022)에 BioStar X를 배포합니다.
- 설치 후 OS 버전을 **검증**합니다.
- 지원 종료 날짜 전에 **업그레이드를 계획**합니다.
- **자동 업데이트를 활성화**하거나 WSUS/SCCM에 연결합니다.
- 지원되지 않는 소프트웨어에서 실행하지 않도록 **라이프사이클을 문서화**합니다.

#### 3.1.2. 프로덕션 배포 전 최신 Windows 업데이트 및 보안 패치 적용

**중요한 이유:**  
패치되지 않은 Windows 서버는 랜섬웨어, 원격 코드 실행, 권한 상승 익스플로잇의 **가장 일반적인 공격 벡터**입니다. 최신 보안 업데이트를 적용하면 BioStar X 호스트가 모든 **알려진 취약점**으로부터 보호된 상태로 라이프사이클을 시작합니다.

**현재 패치 수준 확인**

- `설정 → 업데이트 및 보안 → Windows Update` → "업데이트 확인" 클릭
- 또는 PowerShell에서 최근 설치된 패치 10개를 확인합니다:

```powershell
Get-HotFix | Sort-Object -Descending InstalledOn | Select-Object -First 10
```

> ⚠️ 최신 패치가 30일 이상 된 경우 즉시 업데이트하십시오.

**업데이트 프로세스**

1. Windows Update 수동 실행
2. 모든 보안 및 중요 업데이트 설치 (서버가 아직 프로덕션 상태가 아니더라도 미루지 말 것)
3. 필요시 재부팅
4. "최신 상태입니다"가 표시될 때까지 업데이트 확인 반복

**지속적인 패치 전략**

- **월별 "패치 화요일"**: Microsoft는 매월 두 번째 화요일에 새 패치를 배포합니다.
- 스테이징 환경이 있다면 테스트 후 배포에서 1–2주 이내에 패치를 적용합니다.
- 즉시 패치할 수 없는 경우 **보완적 제어**를 활성화합니다: 방화벽 제한, 인터넷 접근 차단, 패치 전까지 강화 모니터링.

#### 3.1.3. 불필요한 Windows 기능 제거 또는 비활성화

**중요한 이유:**  
활성화된 모든 Windows 기능은 서버의 **공격 표면을 증가**시킵니다. 전용 BioStar X 호스트에서는 Print Spooler, Fax, IIS 등의 서비스가 거의 필요 없으며, 이는 심각한 취약점(예: PrintNightmare)의 이력을 가지고 있습니다.

**단계별 조치:**

1. **서버 관리자** → 관리 → 역할 및 기능 제거
2. 설치된 역할(예: 파일 서비스, IIS, 인쇄 서비스)을 검토하고 BioStar X에 필요하지 않은 역할을 제거합니다.
3. **.NET Framework 기능** 및 관리 워크플로에 필요한 **데스크탑 환경**만 유지합니다.
4. `services.msc`를 열고 불필요한 서비스(Fax, Remote Registry, Bluetooth, SMBv1)를 **사용 안 함**으로 설정합니다.

👍 **모범 사례**: 호스트를 BioStar X **전용**으로 유지합니다. 추가 워크로드 없음.

---

#### 3.1.4. Windows Defender 바이러스 백신 활성화 및 최신 상태 유지

**중요한 이유:**  
Defender(기본 제공 AV)는 악성 코드 및 랜섬웨어로부터 보호합니다. 비활성화하면 호스트가 노출됩니다.

**단계별 조치:**

1. **Windows 보안** → 바이러스 및 위협 방지 열기
2. **실시간 보호**가 활성화되어 있는지 확인
3. **클라우드 기반 보호** 및 **자동 샘플 제출**이 활성화되어 있는지 확인
4. 바이러스 정의를 최신 상태로 유지 (일반적으로 Windows Update를 통해 자동으로 수행)

> ⚠️ 타사 AV를 사용하는 경우 충돌을 방지하기 위해 **하나의 AV 솔루션**만 실행되고 있는지 확인합니다.

---

#### 3.1.5. 물리적 도난 방지를 위한 BitLocker 활성화

**중요한 이유:**  
공격자가 서버에 물리적으로 접근하면(예: 디스크 도난, VM 스냅샷, 백업 미디어) 암호화되지 않은 드라이브에서 자격 증명, 로그, 생체 정보 등 민감한 데이터에 직접 접근할 수 있습니다. BitLocker는 미사용 데이터 보호를 위한 전체 디스크 암호화를 제공합니다.

**권고 사항:**

- 다음에 BitLocker를 활성화합니다:
  - OS 드라이브(C:)
  - BioStar X 데이터베이스, 로그, 백업을 저장하는 모든 데이터 드라이브
- 더 강력한 보호를 위해 가능한 경우 TPM + PIN 사용

**단계별 조치 (GUI):**

1. `제어판 → BitLocker 드라이브 암호화` 열기
2. 각 드라이브에 대해 "BitLocker 켜기" 클릭
3. 인증 방법 선택: TPM(기본값) 또는 TPM + PIN(더 높은 보안에 권장)
4. 복구 키 저장: Active Directory/Azure AD(선호) 또는 보안 볼트(동일 서버에 저장 금지)
5. 암호화 모드: XTS-AES 256비트(권장)
6. 암호화 시작

**모니터링:**

```powershell
Get-BitLockerVolume
```

> ⚠️ BitLocker는 **미사용 데이터만** 보호합니다. 데이터베이스 암호화 또는 TLS 암호화를 대체하지 않습니다.

---

#### 3.1.6. 안전한 인증 정책 시행

**중요한 이유:**  
약한 로컬 계정이나 캐시된 자격 증명은 서버를 무차별 대입 공격이나 권한 상승에 취약하게 만듭니다.

**단계별 조치:**

1. **로컬 보안 정책**(`secpol.msc`) 열기
2. 암호 정책 구성:
   - 최소 길이: 12자 이상
   - 복잡성: 활성화
   - 최대 사용 기간: 90일
   - 잠금 임계값: 5회 시도
3. 기본 "Administrator" 계정 비활성화 또는 이름 변경
4. RDP 또는 원격 관리 접근에 대해 MFA 요구

---

#### 3.1.7. 보안 기준선을 위한 그룹 정책 설정 시행

**중요한 이유:**  
Microsoft의 **Security Compliance Toolkit(SCT)**은 보안 기준선에 맞추어진 다운로드 가능한 **그룹 정책 개체(GPO)**를 제공합니다. 이 기준선은 구성 오류 위험을 줄이고 BioStar X가 강화된 Windows OS에서 실행되도록 합니다.

**단계별 조치:**

1. [Microsoft Security Compliance Toolkit](https://learn.microsoft.com/en-us/windows/security/operating-system-security/device-management/windows-security-configuration-framework/security-compliance-toolkit-10) 다운로드
2. OS 버전에 맞는 기준선 선택 (Windows Server 2019 또는 2022 Security Baseline)
3. **그룹 정책 관리 콘솔(GPMC)**에 기준선 GPO 가져오기
4. BioStar X 서버 OU에 기준선 GPO 연결
5. 중요 설정 검토: 계정 잠금, 감사 로깅, 네트워크 보안(SMB 서명, LDAP 서명), LSA 보호
6. **스테이징에서 먼저 테스트** 후 BioStar X 호환성 확인 후 프로덕션에 배포

---

#### 3.1.8. 필요하지 않은 경우 원격 데스크톱 비활성화(필요한 경우 강화)

**중요한 이유:**  
RDP는 무차별 대입 공격, 자격 증명 도용의 일반적인 공격 벡터입니다.

**필요하지 않은 경우:**
- `시스템 속성 → 원격 → "원격 연결 허용 안 함"`으로 비활성화

**필요한 경우:**
- VPN 또는 LAN으로만 제한
- NLA(네트워크 수준 인증) 요구
- 기본 RDP 포트 변경 (3389 → 사용자 지정)
- 이벤트 로그 및 알림으로 모니터링

### 3.2. 사용자 및 접근 제어

#### 3.2.1. 최소 권한 시행

- Active Directory 또는 로컬 사용자 및 그룹에서 별도의 사용자 그룹 생성
- 그룹 기반으로 파일 시스템 및 애플리케이션 권한 할당
- OS 수준 최소 권한 모델을 반영하도록 BioStar X 내 역할 구성

#### 3.2.2. 로컬 관리 권한 제한

- `lusrmgr.msc` 사용하여 Administrators 그룹에서 불필요한 계정 제거
- 일반 작업에는 표준 사용자 계정 사용, 필요할 때만 관리자 계정 사용
- 도메인에 가입된 경우 그룹 정책 제한 그룹을 적용하여 일관된 권한 시행

#### 3.2.3. 강력한 인증 시행

- 그룹 정책에서 복잡한 암호 요구(최소 12–15자, 대/소문자, 숫자, 기호 혼합)
- 암호 만료(예: 90일마다) 설정 및 재사용 제한
- 여러 번 실패한 시도 후 계정을 잠그는 계정 잠금 정책 구성
- 원격 접근 또는 특권 계정에 MFA 시행

#### 3.2.4. 기본 제공 계정 관리

- `lusrmgr.msc`에서 Administrator 계정을 마우스 오른쪽 단추로 클릭하여 이름 변경
- 명시적으로 필요하지 않은 경우 Guest 계정 비활성화
- PowerShell에서 `net user`로 정기적으로 계정 감사

#### 3.2.5. 세션 관리

- 15분 비활동 후 자동 잠금 구성
- 가능한 경우 동일 계정으로 동시 로그인 방지
- 유휴 원격 세션 제한

그룹 정책 설정 경로: `컴퓨터 구성 → Windows 설정 → 보안 설정 → 로컬 정책 → 보안 옵션 → 대화형 로그온: 컴퓨터 비활동 제한`(예: 900초 = 15분)

#### 3.2.6. 감사 및 모니터링

- 로그인 시도, 권한 사용, 계정 관리에 대한 감사 활성화
- 로그를 **SIEM** 또는 중앙 로그 수집기로 전달
- 이상 여부를 위해 보안 로그 정기적으로 검토

그룹 정책 경로: `컴퓨터 구성 → Windows 설정 → 고급 감사 정책 구성`
활성화 범주: 로그온/로그오프, 계정 관리, 권한 사용

### 3.3. 네트워크 및 방화벽

BioStar X는 HTTPS, REST, gRPC, WebSocket, Thrift를 통해 서비스를 노출할 수 있으므로 인바운드 및 아웃바운드 연결을 엄격하게 제어하는 것이 중요합니다.

#### 3.3.1. Windows Defender 방화벽 활성화 및 구성

**중요한 이유:**  
제대로 구성된 방화벽은 명시적으로 승인된 트래픽만 BioStar X에 도달하도록 하여 공격 표면을 크게 줄입니다.

**BioStar X 기본 네트워크 포트**

| 서비스 | 용도 | 기본 포트 |
|--------|------|----------|
| BioStar X Core Web Service | HTTPS / 웹 인터페이스 | 5002 |
|  | Thrift RPC | 9310 |
| BioStar X Core Service | WebSocket | 9002 |
|  | FastCGI | 9000 |
|  | API | 9010 |
|  | RPC | 51218 |
| Unified Gateway Service | HTTPS | 443 |
| Coordinator Service | 클라이언트 통신 | 21810 |
| Main Server | TCP 서버 | 51212 |
|  | SSL 서버 | 51213 |
|  | gRPC | 51219 |
| Cache Service | 클라이언트 | 10800 |
|  | 통신 | 47500 |
|  | 검색 | 47100 |

> **참고**: Communication Server가 배포된 경우 Main Server 통신 포트(51212, 51219)를 동일하게 사용합니다. 배포 토폴로지에 따라 이 포트가 Communication Server와 Main Server 간에 도달 가능한지 확인하십시오.

**단계별 조치:**

1. `wf.msc` 열기
2. **인바운드 규칙** → 필요하지 않은 규칙 제거 또는 비활성화
3. 다음에 대한 명시적 허용 규칙 생성:
   - Unified Gateway HTTPS(443) – 필요한 경우 외부/공용
   - Core Web Service HTTPS(5002) – 가급적 443을 통해 프록시
   - 내부 전용 포트(gRPC 51219, Cache 10800/47500/47100, Coordinator 21810, Thrift 9310) → 로컬호스트 또는 신뢰할 수 있는 서브넷에서만 허용
4. 다른 모든 인바운드 트래픽은 **차단**
5. **아웃바운드 규칙** → 알려진 신뢰할 수 있는 엔드포인트로만 제한

> ⚠️ 광범위한 "모두 허용" 규칙을 피하십시오. 기본 포트 변경은 약간의 모호성 이점을 제공하지만 방화벽 적용을 대체하지 않습니다.

---

#### 3.3.2. 원격 접근 제한

- RDP를 승인된 관리자로만 제한
- RDP가 필요한 경우: NLA 사용, VPN 또는 점프 호스트 뒤에 배치
- 외부 접근에는 VPN 또는 점프 호스트 요구

---

#### 3.3.3. 안전한 네트워크 세분화

- BioStar X 서버를 사용자 네트워크와 분리된 전용 VLAN 또는 DMZ에 배치
- BioStar X와 필요한 시스템(데이터베이스 서버, API 소비자, 리더) 간의 통신만 허용
- 관리 접근(RDP, SSH)을 관리자 서브넷으로만 제한

👍 **모범 사례**: 세분화와 호스트 방화벽을 결합하여 **심층 방어** 구현.

---

#### 3.3.4. 네트워크 프로토콜 강화

**SMBv1 비활성화:**

```powershell
Disable-WindowsOptionalFeature -Online -FeatureName SMB1Protocol
```

**TLS 1.2+ 시행:**
- 레지스트리에서 SSL 2.0, SSL 3.0, TLS 1.0, TLS 1.1 비활성화
- IIS Crypto 또는 `nmap --script ssl-enum-ciphers`로 검증

> ⚠️ 호환성 문제를 방지하기 위해 프로토콜 비활성화 전에 스테이징에서 테스트합니다.

#### 3.3.5. 네트워크 침입 감지 및 모니터링

- IDS/IPS(Snort, Suricata 또는 상용 제품) 배포
- Windows Defender 방화벽 로깅 활성화
- SIEM으로 로그 전달

방화벽 로그 저장 위치: `%systemroot%\system32\LogFiles\Firewall\pfirewall.log`

### 3.4. 서비스 및 애플리케이션 계층

#### 3.4.1. 서비스에 최소 권한 원칙 적용

- Local System 또는 Administrator 대신 전용 서비스 계정으로 BioStar X 서비스 실행
- 필요한 최소 권한만 부여(서비스로 로그온, 특정 디렉터리 읽기/쓰기)

#### 3.4.2. 애플리케이션 구성 파일 보안

- 비밀 정보(예: 데이터베이스 연결 문자열, API 키)가 포함된 구성 파일에 대한 접근 제한
- 가능한 경우 Windows DPAPI 또는 Key Vault 솔루션 사용
- 바이너리와 별도로 비밀 저장, 하드 코딩 방지

**암호화 키 파일 보호 (`util/` 디렉터리):**

- BioStar X 서비스 계정만 읽기 권한을 갖도록 NTFS 권한 제한
- 표준 사용자 및 대화형 로그인에 대한 접근 거부
- 업그레이드 전에 `util/` 디렉터리를 오프라인 저장소에 안전하게 백업

> ⚠️ 이 파일을 분실하거나 침해당하면 데이터 손실 또는 무단 복호화가 발생할 수 있습니다.

💡 **모범 사례**: 저장된 비밀(DB 암호, API 키)을 주기적으로 교체합니다.

#### 3.4.3. 애플리케이션 런타임 구성 요소 유지

- BioStar X와 종속 런타임 구성 요소(예: Visual C++ Redistributables)를 최신 상태로 유지
- 패치를 위한 벤더 권고 사항 구독
- 가동 중단을 방지하기 위한 업데이트 절차 문서화

#### 3.4.4. 애플리케이션 허용 목록

- WDAC(Windows Defender Application Control) 또는 AppLocker를 사용하여 신뢰할 수 있는 바이너리만 허용
- 임시 디렉터리(예: `%TEMP%`, `%APPDATA%`)에서의 실행 차단
- 먼저 감사 모드에서 배포하여 테스트한 후 적용

### 3.5. 데이터베이스 및 데이터 보호

BioStar X는 배포에 따라 **MariaDB(11.4)** 또는 **Microsoft SQL Server**를 사용합니다.

#### 3.5.1. 인증 및 접근 제어 시행

- 모든 데이터베이스 연결에 인증 요구
- BioStar X에 전용 데이터베이스 계정 사용(공유 또는 기본 계정 사용 금지)
- 필요한 최소 권한 역할만 할당(애플리케이션 데이터에 대한 읽기/쓰기, 스키마 수정 불가)
- 사용하지 않거나 과도한 권한을 가진 사용자를 위해 데이터베이스 계정 정기 감사
- IP 기반 접근 항목에서 오래된 항목 제거

> ⚠️ BioStar X가 sa 또는 동급 슈퍼유저 계정으로 연결되도록 허용하지 마십시오.

---

#### 3.5.2. 전송 중 데이터 암호화

**MariaDB 설정 (`my.cnf`):**

```ini
[mysqld]
require_secure_transport=ON
ssl_cert=/etc/mysql/server-cert.pem
ssl_key=/etc/mysql/server-key.pem
ssl_ca=/etc/mysql/ca-cert.pem
```

검증:
```sql
SHOW VARIABLES LIKE '%ssl%';
```

**SQL Server 설정:** SQL Server 구성 관리자 → MSSQLSERVER 프로토콜 → 암호화 강제 활성화 → 신뢰할 수 있는 인증서 설치

---

#### 3.5.3. 미사용 데이터 암호화

**SQL Server TDE:**

```sql
CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'StrongP@ssw0rd!';
CREATE CERTIFICATE MyServerCert WITH SUBJECT = 'BioStarX';
CREATE DATABASE ENCRYPTION KEY WITH ALGORITHM = AES_256 ENCRYPTION BY SERVER CERTIFICATE MyServerCert;
ALTER DATABASE biostarx_ac SET ENCRYPTION ON;
```

**MariaDB:** `file_key_management` 플러그인 활성화 및 안전한 키파일에 암호화 키 정의

> ⚠️ 성능 오버헤드가 약 3–5% 있습니다. DB를 다른 서버로 이동하거나 복원할 경우 인증서와 키가 필요합니다. BioStar X의 복원 기능은 암호화된 백업을 지원하지 않으므로 수동 처리가 필요합니다.

---

#### 3.5.4. 데이터베이스 감사 활성화

**MariaDB:**

```sql
INSTALL SONAME 'server_audit';
SET GLOBAL server_audit_logging=ON;
SET GLOBAL server_audit_events='CONNECT,QUERY_DML,QUERY_DDL';
```

**SQL Server:**

```sql
CREATE SERVER AUDIT BioStarX_Audit TO FILE (FILEPATH = 'C:\AuditLogs\');
CREATE SERVER AUDIT SPECIFICATION BioStarX_Spec FOR SERVER AUDIT BioStarX_Audit ADD (FAILED_LOGIN_GROUP);
ALTER SERVER AUDIT BioStarX_Audit WITH (STATE = ON);
```

#### 3.5.5. 백업 및 안전한 저장

- BioStar X 데이터베이스를 정기적으로 암호화하여 백업
- 백업은 동일한 호스트가 아닌 별도의 보안 서버에 저장
- 주기적으로 백업 복원 테스트

**SQL Server 암호화 백업:**

```sql
BACKUP DATABASE biostarx_db TO DISK = 'D:\backups\biostarx_db.bak' 
WITH ENCRYPTION (ALGORITHM = AES_256, SERVER CERTIFICATE = MyServerCert);
```

**MariaDB:**

```bash
mariadb-dump --all-databases --ssl --result-file=/secure/backups/biostarx.sql
gpg -c /secure/backups/biostarx.sql
```

> ⚠️ **제품 한계**: BioStar X 백업 기능(수동 또는 예약)은 암호화되지 않습니다. BioStar X 복원 기능은 암호화된 백업을 지원하지 않습니다.

## 4. BioStar X 강화

### 4.1. 시스템 보안

#### 4.1.1. 기본 관리자 로그인 ID 및 암호 변경

**권고 사항:**  
설치 직후 기본 **Administrator** 로그인 ID(admin)를 변경하고 강력하고 고유한 암호를 지정합니다.

**중요한 이유:**  
기본 자격 증명은 공격자의 가장 일반적인 진입점 중 하나입니다. 모든 BioStar X 설치는 사전 정의된 Administrator 계정(UID: 1)과 함께 제공되므로, 변경하지 않으면 무차별 대입 또는 사전 공격에 매우 취약합니다.

**단계별 조치:**

1. 기본 admin 계정으로 로그인
2. **사용자 모듈**로 이동
3. **Administrator** 계정(사용자 이름: Administrator, UID: 1) 더블 클릭
4. **사용자 편집** 페이지의 **권한** 섹션에서:
   - **로그인 ID**: admin에서 고유하고 명확하지 않은 사용자 이름으로 변경 (root, system, 회사명 사용 금지)
   - **암호**: 보안 암호로 변경 (기본적으로 BioStar X는 중간 강도 적용: 8–32자, 문자 및 숫자 포함 필요)
5. **적용** 클릭 후 새 자격 증명으로 다시 로그인하여 확인

**모범 사례:**
- 개인 이름이나 단순한 패턴(admin1, company2025) 사용 금지
- 새 자격 증명을 안전한 암호 관리자 또는 엔터프라이즈 볼트에 저장
- 비상 사용을 위한 **보조 관리자 계정** 생성
- 관리자 로그인 시도에 대한 **감사 로그** 정기적으로 검토

---

#### 4.1.2. 접근 분리

**권고 사항:**  
BioStar X 내에서 **Administrator**, **Operator**, **Auditor** 계정에 명확히 정의된 권한을 부여하여 역할 분리를 엄격하게 시행합니다.

**중요한 이유:**  
모든 작업에 단일 슈퍼유저 계정을 사용하면 불필요한 위험이 생깁니다. 해당 계정이 침해되면 공격자가 보안 설정과 운영 데이터 모두에 대한 완전한 제어권을 얻게 됩니다.

**설치 후 권고 접근 방식:**

1. **기본 Admin 이름 변경 및 보안**: 내장 Administrator 계정의 이름을 변경하고 강력한 암호를 오프라인 볼트에 저장합니다. 일상적인 로그인을 비활성화하고 **"비상 전용" 계정**으로 취급합니다.

2. **명명된 역할 계정 생성**:
   - **Admin 역할**: 시스템 구성 및 역할 관리 전용. 1–2명의 신뢰할 수 있는 사람으로 제한.
   - **Operator 역할**: 일상적인 운영(사용자 추가/제거, 도어/장치 제어). 시스템 보안 설정 또는 감사 로그 접근 없음.
   - **Auditor 역할**: 로그, 보고서, 구성에 대한 읽기 전용 접근. 설정 변경 불가.

3. **맞춤형 계정 수준 사용 (필요한 경우)**:
   - `설정 모듈 → 계정 → 맞춤 수준 추가`로 이동
   - 사용자 그룹, 장치 그룹, 도어 그룹, 엘리베이터 그룹, 접근 그룹에 걸쳐 세부적인 접근 구성
   - 각 모듈에 대해 **읽기 전용** 또는 **편집/읽기** 선택

**맞춤형 Auditor 역할 생성 예시:**

1. `설정 모듈 → 계정 → 맞춤 수준 추가`
2. 이름 = "Auditor", 설명 추가
3. 관리자 메뉴 설정에서:
   - **사용자** = 읽기 전용
   - **모니터링** = 읽기 전용
   - **데이터** = 읽기 전용
   - **대시보드** = 읽기 전용
   - **설정** = 없음
4. **적용** 후 지정된 감사자 계정에 역할 할당

#### 4.1.3. 강력한 암호 정책 시행

**경로:** `설정 → 시스템 → 보안 → 로그인 암호`

**암호 정책 옵션:**

- **암호 수준**: 낮음(복잡성 없음) / **중간(기본값)**: 8–32자, 문자 및 숫자 / **강력**: 10–32자, 대소문자, 숫자, 기호
- **최대 암호 사용 기간**: 1–180일(만료 시 변경 요구)
- **최대 잘못된 시도 횟수**: 지정된 시간 내 실패 횟수 후 잠금(기본값: 10분 내 100회 → **15분 내 5–10회로 줄이는 것 권장**)
- **최대 암호 변경 제한**: 암호를 빠르게 변경하여 암호 이력 정책을 우회하는 것을 방지

**권장 구성:**
- 암호 수준: **강력**
- 최대 암호 사용 기간: **90일**
- 최대 잘못된 시도: **15분 내 5회**
- 최대 암호 변경 제한: **5회**

#### 4.1.4. 세션 보안

**경로:** `설정 → 시스템 → 보안 → 세션 보안`

**동시 연결 허용**을 **비활성화**합니다. 이렇게 하면 각 계정에 하나의 활성 세션만 허용됩니다.

#### 4.1.5. 다단계 인증 (MFA)

**권고 사항:**  
지문 스캐너를 통한 추가 생체 인식 요소를 요구하여 모든 고권한 계정에 MFA를 활성화합니다.

**BioStar X에서 작동 방식:**
- MFA는 지문 스캐너(Suprema BioMini 또는 BioMini Plus 2)를 사용하여 구현
- 사용자가 로그인 ID와 암호를 입력하면 시스템이 지문 스캔을 요청
- 두 요소가 모두 성공해야 로그인 완료

> ⚠️ 주의 사항:
> - MFA가 활성화된 상태에서 지문 구성 요소가 실패하면 잠길 수 있습니다
> - 클라우드 인터페이스를 통한 로그인 시 MFA는 지원되지 않습니다
> - Active Directory 또는 지원되지 않는 서비스를 통해 인증된 계정에는 MFA를 활성화할 수 없습니다

**단계별 조치:**
1. BioStar X 서버 또는 클라이언트 머신에 지문 스캐너 설치
2. 사용자 모듈에서 사용자 프로필을 열고 지문 등록
3. 권한 섹션에서 MFA(로그인을 위한 다단계 인증)를 **사용**으로 설정
4. 변경 사항 적용 후 테스트

**모범 사례:**
- 스캐너 오류를 대비하여 예비 지문(다른 손가락) 등록
- 내장 관리자("비상" 계정)는 생체 장치 실패 시 폴백 복구 방법이 있어야 합니다

#### 4.1.6. 장치와의 보안 통신

**권고 사항:**  
디지털 인증서(TLS/SSL 방식)를 사용하여 BioStar X와 연결된 장치 간 트래픽을 암호화하도록 **장치와의 보안 통신**을 활성화합니다.

**경로:** `설정 → 시스템 → 보안 → 고급 보안 설정 → 장치와의 보안 통신`

**옵션 A — 서버 생성 인증서(기본값):**
- 간편한 설정: BioStar X가 인증서를 생성하여 장치에 자동 프로비저닝
- 한계: 이 설정을 비활성화했다가 다시 활성화하면 새 인증서가 생성되어 수동 장치 재설정이 필요할 수 있음

**옵션 B — 외부 인증서(엔터프라이즈 사용 권장):**
- 조직의 인증서 체인 및 개인 키 업로드
- 엔터프라이즈 PKI와의 통합 용이

**단계별 조치:**
1. 관리자로 로그인
2. `설정 → 시스템 → 보안 → 고급 보안 설정`
3. **장치와의 보안 통신 = 사용** 설정
4. 외부 CA 인증서를 사용하는 경우: 루트 CA, 공개 인증서, 개인 키 업로드
5. 적용 후 장치가 보안 채널로 다시 연결되는지 확인

> 🔎 **로그 업로드 참고**: **자동 로그 업로드**가 비활성화된 경우 로그를 정기적으로 수집하지 않으면 이벤트 가시성이 지연되거나 감사 추적에 공백이 생길 수 있습니다. 가능한 경우 **자동 로그 업로드를 활성화**하는 것을 권장합니다.

---

#### 4.1.7. 장치 해시키 관리

**권고 사항:**  
BioStar X와 장치 간에 사용되는 암호화 키를 수동으로 제어하도록 **장치 해시키 관리**를 활성화합니다.

**중요한 이유:**  
장치와의 보안 통신(4.1.6)이 인증서를 사용하여 전송 중 데이터를 보호하는 반면, 장치 해시키 관리는 한 단계 더 나아갑니다:
- **데이터 암호화 키가 관리자에 의해 제어**되도록 보장합니다
- **탬퍼 방어**를 활성화합니다 — 장치가 무단 개방이나 조작을 감지하면 저장된 사용자, 로그, 암호화 키를 삭제합니다

**경로:** `설정 → 시스템 → 보안 → 고급 보안 설정 → 장치 해시키 관리`

> ⚠️ 중요 참고 사항: 비관리자 사용자에게 로그인 PW 또는 PIN이 있는 경우 수동 키 관리를 적용할 수 없습니다. 활성화하기 전에 이러한 로그인을 제거하십시오.

**모범 사례:**
- 장치 해시키를 정기적으로 교체(예: 연간 또는 반기별)
- 모든 교체를 문서화하고 안전한 볼트에 저장
- 랩/스테이징 장치에서 탬퍼 이벤트 테스트
- **장치와의 보안 통신(4.1.6)**과 함께 사용하여 심층 방어 달성

#### 4.1.8. 시스템 백업

**권고 사항:**  
내장 **시스템 백업** 기능을 사용하여 BioStar X의 데이터베이스, 시스템 설정, 키 값을 정기적으로 백업합니다.

**경로:** `설정 → 시스템 → 시스템 백업`

> ⚠️ **제품 한계**:
> - BioStar X와 MSSQL이 다른 서버에 설치된 경우 시스템 백업/복원을 사용할 수 없습니다. 이 경우 MSSQL 기본 백업 절차를 사용하십시오.
> - BioStar X 백업은 암호화되지 않습니다.
> - BioStar X 복원 기능은 암호화된 백업을 지원하지 않습니다.

**일반 설정:**
- **백업 파일 경로**: 보안 접근 제어 폴더 지정 (예: `C:\Users\Public` 사용 금지)
- **유지할 백업 파일 수**: 일일 예약에는 최소 7–10개 권장
- **지금 백업**: 업그레이드, 패치, 주요 구성 변경 전 즉시 백업 실행에 유용

**자동 시스템 백업:**
- 빈도: 일별 / 주별 / 월별
- 프로덕션 배포에는 **오프피크 시간(예: 오전 2시)에 일별 백업** 권장

**자동 백업 구성 단계:**
1. `설정 → 시스템 → 시스템 백업`
2. 자동 시스템 백업: 빈도 = 일별, 시간 = 02:00, 유지할 파일 수 = 30
3. 적용 후 대상 디렉터리에서 백업 생성 확인
4. 스테이징 환경에서 주기적으로 **복원 절차 테스트**

> ⚠️ 백업/복원 중 서버 전원이 중단되지 않도록 UPS로 보호하십시오. 중단 시 데이터베이스 손상이 발생할 수 있습니다.

#### 4.1.9. 디렉터리 통합

디렉터리 통합을 통해 BioStar X를 **Microsoft Active Directory(AD)** 또는 **Entra ID(Azure AD)** 같은 엔터프라이즈 ID 시스템과 연결할 수 있습니다.

> ⚠️ **라이선스 참고**: 디렉터리 통합에는 **Advance 라이선스**가 필요합니다.

**활성화 권장 시나리오:**
- IT 시스템에 AD/Entra ID를 이미 사용하는 중대형 기업
- 사용자 수가 적고 BioStar X에서 직접 계정을 관리할 수 있는 소규모 배포에는 선택 사항

**Microsoft Active Directory 설정:**

1. BioStar X에 관리자로 로그인
2. `설정 → 디렉터리 통합`
3. **디렉터리 서비스** = Microsoft Active Directory 선택
4. 서버 주소, 사용자 이름, 암호, 기본 도메인 이름 구성
5. LDAPS(SSL/TLS를 통한 LDAP)를 위해 루트 인증서를 .jks로 변환하여 BioStar X 설치 경로에 배치
6. **연결**을 클릭하고 AD 사용자 그룹 가져오기
7. 필요하지 않은 그룹 선택 해제
8. **Active Directory로 로그인 활성화** 설정
9. 동기화 옵션 구성: 수동 또는 자동(30분–7일)

**Microsoft Entra ID 설정:**

1. BioStar X에 관리자로 로그인
2. `설정 → 디렉터리 통합`
3. **디렉터리 서비스** = Microsoft Entra ID 선택
4. 클라이언트 ID, 클라이언트 암호, 기본 도메인(테넌트 ID) 구성
5. **연결**을 클릭하고 사용자 그룹 정보 가져오기
6. 관련 그룹만 선택하여 동기화
7. **Entra ID로 로그인 활성화**
8. BioStar X의 **리디렉션 URI**를 복사하여 Entra ID SSO 설정에 붙여넣기

**모범 사례:**
- 제한된 권한을 가진 **전용 AD/Entra 서비스 계정** 사용
- Active Directory 연결에는 항상 **LDAPS** 사용
- 자동 동기화를 최소 일별로 설정
- BioStar X에서 동기화된 계정을 주기적으로 검토하여 오래된 항목 확인

#### 4.1.10. 시스템 로그 설정

**경로:** `설정 → 서버 → 서버`

**시스템 로그 저장 기간:**
- **자동(기본값)**: 보존 기간 후 로그 자동 삭제
- **수동**: 수동으로 삭제
- 보존 기간: 1–120일(기본값: 60일)

💡 컴플라이언스 민감 환경(금융, 정부)에서는 최대치에 가깝게 설정(90–120일)하고 장기 저장을 위해 SIEM으로 전달합니다.

**시스템 로그 수준 설정:**

| 수준 | 설명 |
|------|------|
| Trace | 모든 것 캡처(매우 상세) |
| Debug | 상세 개발자 수준 로그 |
| Info | 운영 이벤트, 성공한 연결(대부분의 기본값) |
| Warning | 잠재적 문제 |
| Error | 실패만 |

> ⚠️ 프로덕션에서 **Trace/Debug**를 실행하면 과도한 로그가 생성되어 성능에 영향을 미치고 디스크를 빠르게 채울 수 있습니다. 단기 문제 해결에만 사용합니다.

**BioStar X 기본 로그 수준:**
- Debug, SQL, WebSocket: 사용 안 함
- Network, System, Web: Info

#### 4.1.11. 서버 포트 구성

**권고 사항:**  
BioStar X 포트를 명시적으로 구성하고 가능한 경우 보안 채널을 선호합니다. 시스템 접근에는 기본 HTTPS 포트(443)를 사용하고, 장치 통신 포트는 신뢰할 수 있는 네트워크로만 제한합니다.

**HTTPS 포트 변경 단계:**

1. **BioStar X 서비스 관리자** 열기: `http://127.0.0.1:28088/`
2. 측면 메뉴에서 **서비스 설정** 선택
3. **Unified Gateway Service**에서 **HTTPS 포트** 필드(기본값: 443) 찾기
4. 원하는 값(예: 8443)으로 변경
5. **적용** 후 필요시 서비스 재시작
6. 방화벽 규칙 및 클라이언트 연결 업데이트

### 4.2. 개인 데이터 보안

#### 4.2.1. 데이터베이스의 개인 데이터 암호화

**권고 사항:**  
모든 민감한 사용자 및 자격 증명 정보가 암호화된 상태로 저장되도록 **데이터베이스의 개인 데이터 암호화**를 활성화합니다.

**경로:** `설정 → 시스템 → 보안 → 데이터베이스의 개인 데이터 암호화`

**암호화되는 데이터 범위:**
- 프로필 이미지
- 사용자 ID, 이름, 전화번호, 사용자 IP, 이메일 주소
- 로그인 ID 및 로그인 암호
- 얼굴 템플릿, 지문 템플릿
- 카드 ID, 스마트 카드 레이아웃 키
- 사용자 및 방문자를 위한 맞춤형 정보
- 이미지 로그 파일

**개인 데이터 암호화 키:**
- 암호화를 **사용**으로 설정하면 구성 가능
- 키는 **32자**여야 하며 문자, 숫자, 기호 포함
- 키를 변경하면 모든 기존 데이터의 재암호화가 발생

**단계별 조치:**
1. 관리자로 로그인
2. `설정 → 시스템 → 보안`
3. **데이터베이스의 개인 데이터 암호화 = 사용** 설정
4. 선택적으로 **개인 데이터 암호화 키** 아래 **변경** 클릭 → 새 32자 키 입력

> ⚠️ 암호화 키 변경 시 모든 기존 개인 데이터가 재암호화됩니다(데이터베이스 크기에 따라 시간이 걸릴 수 있습니다). 유지 관리 기간에 예약하십시오. 키를 분실하면 암호화된 데이터를 복구할 수 없을 수 있습니다.

#### 4.2.2. 시각적 얼굴 이미지 저장

**권고 사항:**  
엄격한 운영 요구 사항이 없는 한 **시각적 얼굴 이미지 저장**을 비활성화합니다. 기본적으로 BioStar X는 **시각적 이미지**와 **얼굴 템플릿**(특징 추출된 생체 데이터)을 모두 저장합니다. 이 옵션을 비활성화하면 **템플릿**만 보관되어 개인 정보 위험이 줄어듭니다.

**중요한 이유:**
- **템플릿 대 이미지**: 템플릿은 생체 데이터의 수학적 표현으로 실제 얼굴로 역공학이 어렵습니다. 반면 이미지는 직접 식별 가능합니다.
- 많은 개인 정보 보호 규정(GDPR, PDPA 등)이 생체 이미지를 매우 민감한 데이터로 취급합니다.

**경로:**
1. `설정 → 서버`
2. **전문가 설정 표시 = 사용** 활성화
3. **전문가 설정**에서 **시각적 얼굴 이미지 저장** 찾기
4. 기본값: **사용** → 권장: **사용 안 함**으로 설정

**모범 사례:**
- 생체 인증에는 이미지가 아닌 **템플릿**만 보관
- 조사 또는 감사 목적으로 이미지가 필요한 경우: 별도의 보안 저장소에 보관하고 짧은 보존 기간(30–60일) 적용
- 업그레이드 후 설정이 기본값으로 복원되지 않았는지 주기적으로 검토

## 5. Suprema 장치 강화

본 섹션은 BioStar X에 연결된 Suprema 제조 접근 제어 장치(생체 인식 단말기, RFID 리더, 컨트롤러 등)에 대한 강화 권고 사항을 제공합니다.

**범위:**
- BioStar X에 연결된 Suprema 접근 제어 장치에 적용
- 펌웨어 유지 관리, 장치 이벤트 로그 처리, 관리자 구성, 장치 측 운영 제어 포함
- 장치 기능은 모델 및 펌웨어 버전에 따라 다를 수 있으며, 구성 변경 적용 전 항상 제품별 장치 매뉴얼 및 릴리스 노트를 참조하십시오.

### 5.1. 장치 펌웨어 최신 상태 유지

**권고 사항:**  
고객 환경에 대해 승인된 최신 지원 버전으로 Suprema 장치 펌웨어를 최신 상태로 유지합니다.

**일반 모범 사례:**
- 모든 연결된 장치의 인벤토리 유지(모델명, 장치 ID, 펌웨어 버전, 설치 위치, 역할)
- 업그레이드 전 Suprema 펌웨어 릴리스 노트 검토
- 광범위한 배포 전에 비프로덕션 장치에서 펌웨어 업그레이드 테스트
- 유지 관리 기간에 펌웨어 업그레이드 예약
- 업그레이드 중 안정적인 전원 공급 보장
- 업그레이드 후 장치가 BioStar X에 다시 연결되고 사용자, 도어, 이벤트, 인증 동작이 정상인지 확인

#### 5.1.1. 소프트웨어에서 펌웨어 업그레이드

**권장 접근 방식:**

1. 대상 장치 모델에 맞는 펌웨어 파일 확보
2. 릴리스 노트 및 호환성 참고 사항 검토
3. 장치가 온라인이며 BioStar X와 정상 통신 중인지 확인
4. 사용자, 자격 증명, 로그, 장치 구성 동기화 확인
5. 대상 장치 선택 후 펌웨어 업그레이드 실행
6. 업그레이드 완료 및 장치 재시작 후 펌웨어 버전 확인
7. 검증: BioStar X 연결, 인증, 도어 제어, 이벤트 로그 업로드 정상 여부 확인

> ⚠️ 다른 장치 모델용 펌웨어 적용 금지. 업그레이드 중 중단 금지. 여러 중요 출입구 장치를 동시에 업그레이드하지 마십시오.

#### 5.1.2. 장치/USB에서 직접 펌웨어 업그레이드

1. USB 메모리에 펌웨어 파일 저장 (하나의 펌웨어 파일만 유지)
2. USB를 장치에 연결
3. 관리자 수준 자격 증명으로 인증
4. 펌웨어 파일이 장치로 전송될 때까지 대기
5. 장치가 지시할 때 USB 제거
6. 펌웨어 업그레이드 완료 및 자동 재시작 대기
7. 펌웨어 버전 확인 및 장치 동작 검증

> ⚠️ USB를 통한 펌웨어 업그레이드를 진행하기 전에 장치 관리자가 구성되어 있어야 합니다. 업그레이드 중 전원 공급을 끊지 마십시오.

### 5.2. 장치 이벤트 로그 보존 및 보호

**권고 사항:**  
장치 이벤트 로그를 보존하고 BioStar X에 정기적으로 업로드되도록 합니다.

**장치 로그 정책:**
- 관리자가 삭제하거나 용량 한계로 덮어쓰이기 전까지 장치에 로컬로 보존
- 장기 감사 보존을 위해 장치 로컬 저장소에만 의존하지 마십시오
- 가능한 경우 BioStar X로의 자동 로그 업로드 활성화
- 수동 로그 업로드 사용 시 정기적인 로그 수집 일정 정의

**보안 탬퍼 고려 사항:**  
일부 Suprema 장치는 **보안 탬퍼** 동작을 지원합니다. 탬퍼 이벤트 발생 시 장치는 사용자, 로그, 암호화 키, SSL 인증서를 포함한 민감한 데이터를 삭제할 수 있습니다. 따라서 자동 로그 업로드를 활성화하여 탬퍼 전 로그를 BioStar X에 안전하게 저장하는 것이 중요합니다.

### 5.3. 장치 관리자 구성

**권고 사항:**  
지원되는 Suprema 장치에 장치 관리자를 구성하고 장치 관리자 메뉴에 대한 접근을 제한합니다. 장치는 관리자 보호 없이 프로덕션에 배포해서는 안 됩니다.

> 장치 관리자 권한은 BioStar X 사용자 권한과 별개입니다. 사용자를 장치 관리자로 지정하면 해당 사용자가 장치에서 로컬로 수행할 수 있는 작업을 제어하지만, 사용자의 BioStar X 애플리케이션 권한을 부여하거나 수정하지 않습니다.

**일반 모범 사례:**
- 프로덕션 배포 전 최소 한 명의 장치 관리자 등록
- 승인된 직원에게만 할당된 고유한 관리자 자격 증명 사용
- 강력한 인증 방법 사용(카드 + PIN, 생체 + PIN 등)
- 직원 역할이 변경되면 즉시 관리자 접근 제거

#### 5.3.1. 소프트웨어에서 장치 관리자 구성 (권장)

1. BioStar X에 관리자로 로그인
2. 장치 관리자 역할을 할 사용자 생성 또는 선택
3. 장치 기능에 따라 필요한 자격 증명(카드, PIN, 지문, 얼굴) 등록
4. 관리자 수준 권한 할당
5. 대상 장치에 사용자 전송 또는 동기화
6. 장치에서 관리자 자격 증명이 관리자 메뉴에 접근할 수 있는지 확인

#### 5.3.2. 장치에서 직접 장치 관리자 구성

1. 장치 메뉴 접근
2. 사용자 등록 또는 관리자 구성 메뉴 열기
3. 관리자 사용자 및 자격 증명 등록
4. 관리자 수준 권한 할당
5. 설정 저장
6. 관리자 메뉴에 다시 접근하여 관리자 인증 테스트

---

### 5.4. 장치 네트워크 및 로컬 서비스 강화

**권고 사항:**  
최소 기능 원칙에 따라 장치 네트워크 및 로컬 서비스 설정을 구성합니다.

**권장 조치:**
- 안정적이고 문서화된 장치 IP 주소 지정 사용 (DHCP 사용 시 IP 예약, 고정 IP 사용 시 IP 주소, 서브넷 마스크, 게이트웨이, DNS 서버, 장치 포트, 장치 ID, 설치 위치 문서화)
- 필요하지 않은 장치 측 기능 비활성화(RTSP, IP 인터폰 등)
- RTSP, 인터폰 등이 필요한 경우 강력한 자격 증명 구성 및 신뢰할 수 있는 시스템으로만 접근 제한
- 지원되는 경우 시간 동기화 활성화
- 기본값 복원 또는 공장 초기화 후 네트워크 설정, 관리자/운영자 수준, 루트 인증서, 보안 통신, BioStar X 연결 재확인

---

### 5.5. 장치 측 운영 강화 체크리스트

- ✅ 최신 지원 버전으로 펌웨어 업데이트 유지
- ✅ 장치 모델, 장치 ID, 펌웨어 버전, 위치, 역할의 인벤토리 유지
- ✅ 프로덕션 사용 전 장치 관리자 접근 구성
- ✅ 승인된 직원만으로 관리자 권한 제한
- ✅ 지원되는 경우 BioStar X와의 보안 통신 활성화
- ✅ 배포 위험 수준에 맞게 장치 해시키 관리/보안 탬퍼 활성화
- ✅ 가능한 경우 자동 로그 업로드 활성화
- ✅ 장치 이벤트 로그가 BioStar X에 정기적으로 업로드되는지 확인
- ✅ 장치 설치 및 케이블링을 물리적으로 보안
- ✅ 탬퍼 이벤트, 예상치 못한 재시작, 펌웨어 변경, 로그 공백을 보안 관련 이벤트로 취급
- ✅ 펌웨어 업그레이드, 장치 교체, 복원 작업 후 장치 구성 검토
- ✅ 기본값 복원 및 공장 초기화 작업을 승인된 관리자만으로 제한

### 6.2. BioStar X의 기본 보안 기능

| 기능 | UI 위치 | 기본값 | 권장 사항 |
|------|---------|--------|----------|
| 기본 Admin 로그인 ID & 암호 변경 | 사용자 모듈 → 기본 Admin 편집(UID:1) | admin / 설치 PW | 설치 직후 변경. |
| 접근 분리(역할) | 설정 → 사용자 → 계정 수준 | 사전 정의: Admin, Operator, Monitoring | 필요에 따라 맞춤 역할 생성. |
| 암호 강도 정책 | 설정 → 시스템 → 보안 | 중간 | 강력으로 설정하고 MFA 시행. |
| 세션 보안(동시 연결) | 설정 → 시스템 → 보안 | 허용 | 공유 로그인 방지를 위해 **비활성** 설정. |
| 다단계 인증 | 설정 → 시스템 → 보안 → MFA | 사용 안 함 | Admin/Operator/Auditor 계정에 활성화. |
| 장치와의 보안 통신 | 설정 → 시스템 → 보안 → 고급 | 사용 안 함 | 활성화 + SSL 포트(51213) 사용. |
| 장치 해시키 관리 | 설정 → 시스템 → 보안 → 고급 | 사용 안 함 | 고보안 배포에서 탬퍼 보호를 위해 활성화. |
| 시스템 백업 | 설정 → 시스템 → 시스템 백업 | 수동 | 자동 일별 백업 활성화. |
| 디렉터리 통합 | 설정 → 디렉터리 통합 | 구성되지 않음 | 가능한 경우 Entra ID 또는 AD와 통합. |
| 로그 설정(보존 및 수준) | 설정 → 서버 → 서버 | 60일, Info | 컴플라이언스에 맞게 조정. SIEM으로 전달. |
| 데이터베이스의 개인 데이터 암호화 | 설정 → 시스템 → 보안 | 사용 안 함 | 프로덕션에서 활성화. |
| 시각적 얼굴 이미지 저장 | 설정 → 서버 → 전문가 설정 | 사용 | 정당한 이유가 없는 한 비활성화. |
